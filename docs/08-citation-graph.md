# 08. 引用関係グラフ

## 1. データソース

| 優先度 | ソース | 用途 |
|---|---|---|
| 一次 | Semantic Scholar Graph API | `/paper/{id}/references`・`/paper/{id}/citations`・`/paper/batch` |
| **補完（常時マージ）** | OpenAlex | `cites:{id}` フィルタで被引用、`referenced_works` で参考文献を取得し、S2の結果に**外部IDで重複排除しながら統合**する |

OpenAlexを障害時のフォールバックではなく**常時の補完**とする理由（実測に基づく設計判断）:

- S2の引用索引はカバレッジが不完全なことがある（実例: 被引用9件の論文をS2は4件しか把握していなかった。Crossref / OpenAlexは9件で一致）
- S2は**出版社の意向でreferencesを非公開**にすることがある（AIP・Elsevierで実例）。OpenAlexの `referenced_works` はこの場合も取得できる
- マージ規則: S2の結果は従来どおり差し替え、OpenAlex分は**既存エッジを上書きしない追加**（`INSERT OR IGNORE`。エッジの `source` 列は先に取得した側を保持）。S2が失敗してもOpenAlexだけで成立する（逆も同様）

- S2の論文IDは `s2_paper_id` のほか `DOI:...` / `ARXIV:...` プレフィックスで指定可能。取り込み済み論文は通常どれかを持つ
- `fields` パラメータで必要列のみ取得する: `title,year,venue,authors,externalIds,citationCount`
- **APIキーなしでも動作する**（共有レートリミット）。設定画面でS2 APIキーの登録を推奨案内し、キーがあれば `x-api-key` ヘッダで送る
- 取得元は `citations.source`（`s2` | `openalex`）に記録する（→ [02](02-data-model.md)）

## 2. 取得タイミング

1. **論文取り込み完了時**: パイプライン（→ [04](04-ingest-pipeline.md)）の最終ステージ後、`kind = 'refetch_citations'` のジョブを自動投入し references / citations を取得
2. **手動更新**: 論文詳細パネルの「引用情報を更新」で同ジョブを投入
3. **TTL失効時**: グラフ表示時に `fetched_at` が古いエッジを検出し、バックグラウンドで再取得（→ 3節）

取得処理はすべて jobs キュー経由でアプリ（JobRunner）が実行する。

## 3. キャッシュ: citations + stub論文

引用先・引用元の論文は大半がライブラリ外のため、**stub論文**（`papers.is_stub = 1`、書誌のみのDB行）としてキャッシュする。

- エッジは `citations(citing_id, cited_id)` に保存。両端は `papers.id` を参照するため、ライブラリ外の論文もstub行として先にINSERTする
- stub行は `title` / `year` / `venue` / 著者 / `s2_paper_id` / `citationCount` 相当の最小書誌のみ持ち、`meta.json` を持たない（再構築時はAPIから再取得 → [03](03-library-layout.md)）
- 同一論文のstubは外部ID（`s2_paper_id` / `doi` / `arxiv_id` / `openalex_id`）で重複排除し、既存行に追記する

### TTL

- `citations.fetched_at` で管理。**既定TTL 30日**
- グラフ表示時、中心論文のエッジの `fetched_at` が TTL を超過していたら、**キャッシュで即座に描画した上で**バックグラウンド再取得し、完了後にグラフを差分更新する

### 被引用数の多い論文への対策

被引用数が数万件の論文（例: 古典的論文）で citations の全取得は現実的でない。

- citations の取得は**上限 1,000 件**（`citationCount` 降順）。references は通常数十〜数百件のため全件取得
- 上限到達時は `papers` 行に総被引用数を保持し、グラフ上に「+12,345 citations（上位1,000件のみ表示）」と表示

## 4. stub論文の昇格

stubノードの詳細パネルから「**ライブラリに取り込む**」を実行すると:

1. `is_stub = 0` に更新（**同一行のまま昇格**。`papers.id` は変わらず、既存の citations エッジがそのまま有効）
2. `status` 遷移はそのまま取り込みパイプライン（→ [04](04-ingest-pipeline.md)）に乗り、PDF取得 → 変換 → インデックス化が走る
3. 昇格時に `meta.json` を書き出し、ファイル正本の対象になる

逆方向（取り込みが先、stubと後から一致）の場合: ローカルPDF取り込みの書誌解決が既存stubと同じDOIに
確定したときは、**stub行を取り込み行へ吸収**する（エッジ付け替え + stub行削除 → [04](04-ingest-pipeline.md) 4節）。

## 5. 表示（v1）: エゴネットワーク

v1は**選択論文を中心とした1〜2ホップのエゴネットワーク**のみ。ライブラリ全体のグラフはv2（→ [10](10-roadmap-risks.md)）。

### 描画

- SwiftUI `Canvas` + force-directed レイアウト（Fruchterman-Reingold系）の**自前実装**。対象は数百ノード規模であり、外部グラフライブラリへの依存を避ける
- レイアウト計算はバックグラウンドで反復し、収束までアニメーション表示

### ノード・エッジの表現

| 要素 | 表現 |
|---|---|
| ライブラリ内論文（`is_stub = 0`） | 濃色ノード |
| stub論文（`is_stub = 1`） | 薄色（半透明）ノード |
| **中心との関係による色分け** | 参考文献（中心が引用 → 青系）/ 被引用（中心を引用 → 緑系）/ 双方向（相互引用 → 紫系）/ 2ホップ先（間接 → 灰色）。**凡例**をグラフ内に常時表示する |
| ノードサイズ | 被引用数の対数スケール（`log10(citationCount + 1)`） |
| エッジ方向 | 引用方向（citing → cited）。矢印付き |
| 中心論文 | 強調枠 + 固定位置（初期レイアウトの中央） |

### クリック動作

- ノードクリックで詳細パネルを表示（タイトル・著者・年・venue・被引用数）
- ライブラリ内ノード: 「開く」でその論文へ移動（グラフの中心を切り替え可能）
- stubノード: 「ライブラリに取り込む」ボタン（→ 4節）

## 6. インタラクション

| 操作 | 動作 |
|---|---|
| ズーム / パン | ピンチ・スクロール・ドラッグ |
| ホップ数切替 | 1ホップ（直接の引用・被引用）/ 2ホップ。既定は1 |
| **関係フィルタ** | 「すべて / 参考文献（この論文が引用）/ 被引用（この論文を引用）」の切替。参考文献モードは中心から引用方向（citing → cited）に到達可能な部分グラフのみ、被引用モードは逆方向に到達可能な部分グラフのみを表示する（2ホップにも一貫して適用） |
| 年代フィルタ | スライダで表示する出版年範囲を絞り込み |

### 表示ノード数の制限（ハブ論文対策）

被引用の膨大な論文（古典論文等。実測で次数1,600超）をそのまま展開すると、
ノード取得とforce-directedレイアウト（全ペアO(n²)）でメモリ・CPUが爆発しフリーズする。
表示段に以下の上限を設ける（**取得段のAPI上限1,000件とは独立**）:

| 上限 | 既定値 | 動作 |
|---|---|---|
| 1ホップ目のエッジ数 | 150 | 超過分は間引き、中心ノードに「**+N件省略**」と表示する |
| 2ホップ目のエッジ数/ノード | 50 | 同様にノード単位で間引く |
| 全体ノード数 | 400 | 到達した時点で2ホップ目の展開を打ち切る |

自著被引用ネットワーク（→ [09](09-ui.md) 4.1節）にも同じ上限を適用する。
統計ヘッダの件数は**間引き前の真値**を表示する（省略は描画のみ）。

### レイアウト実行

レイアウト計算は**UIスレッドで同期実行しない**。フレームごとに数イテレーションずつ進める
漸進方式とし、収束までアニメーション表示する（5節の描画方針と統一）。
